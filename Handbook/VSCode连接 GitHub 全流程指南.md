# 🚀 VS Code 丝滑连接 GitHub 全流程指南 (通用版)

> **导读：** 在现代软件开发中，版本控制是核心。VS Code 凭借其强大的内置 Git 支持，可以让开发者无需频繁切换工具，即可完成从本地代码到 GitHub 开源的全过程。

------

## 第一步：环境基石 —— Git 检查

VS Code 是一个强大的编辑器，而 Git 是底层的引擎。

1. **打开终端：** 使用快捷键 `Ctrl + ` `。
2. **验证安装：** 输入 `git --version`。
   - 如果返回版本号（如 `git version 2.x.x`），说明环境就绪。
   - 若未安装，请前往 [Git 官网](https://git-scm.com/) 下载。

------

## 第二步：代码仓库初始化 (Initialize)

当你准备好一个项目文件夹，并想让它成为一个受管理的仓库时：

1. **进入面板：** 点击左侧活动栏的 **“源代码管理”** 图标（快捷键 `Ctrl + Shift + G`）。
2. **一键初始化：** 点击 **“初始化存储库” (Initialize Repository)**。
   - **原理：** 此时 Git 会在当前目录下创建一个隐藏的 `.git` 文件夹，开始追踪所有文件的变动。

------

## 第三步：暂存与提交 (Stage & Commit)

这是将改动写入历史记录的关键步骤。

1. **暂存 (Stage)：** 在文件列表旁点击 **`+`** 号。这代表你已经确认这些改动是“准备好发货”的。
2. **提交 (Commit)：**
   - 在顶部的输入框中输入描述（例如：`initial commit` 或 `feat: add core logic`）。
   - **💡 技巧：** 如果安装了 GitHub Copilot，点击输入框旁的 ✨ 按钮，AI 会根据你的代码变动自动生成专业的提交信息。
3. **确认：** 点击蓝色的 **“提交” (Commit)** 按钮，改动即被记录在本地历史中。

------

## 第四步：同步至 GitHub (Publish & Push)

要让全球开发者看到你的代码，你需要将其推送到远程服务器。

### 方案 A：VS Code 自动同步（最快）

1. 提交完成后，点击蓝色的 **“发布分支” (Publish Branch)** 按钮。
2. 在弹出的输入框中选择：
   - **Publish to GitHub public repository** (公开：任何人可见，开源项目首选)。
   - **Publish to GitHub private repository** (私有：仅自己或授权者可见)。
3. **结果：** VS Code 会自动在你的 GitHub 账号下创建同名仓库并完成推送。

### 方案 B：手动关联远程库（最稳）

如果你已经在 GitHub 网页端手动创建了仓库，在终端输入以下“连招”：

1. **关联：** `git remote add origin https://github.com/你的用户名/仓库名.git`
2. **推送：** `git push -u origin main`

------

## 🛠️ 开源开发者必备“发货”连招

在日常维护开源项目时，熟练使用这三行命令或对应的 UI 按钮，能让协作更高效：

| **步骤**    | **终端命令**          | **VS Code UI 操作**               |
| ----------- | --------------------- | --------------------------------- |
| **1. 揽件** | `git add .`           | 点击 Changes 旁的 **+**           |
| **2. 打包** | `git commit -m "msg"` | 输入框写字 + 点击 **Commit**      |
| **3. 出库** | `git push`            | 点击底部状态栏的 **同步循环图标** |

------

## 💡 给开源新手的进阶建议

1. **.gitignore 是必须的：** 在推送前，创建一个 `.gitignore` 文件。
   - 明确哪些文件**不应该**上传（如编译生成的二进制文件、含密钥的配置、本地环境配置等）。
2. **编写 README：** 好的开源项目 50% 的魅力来自文档。
3. **分支规范：** 保持 `main` 分支的稳定性，新功能建议在 `dev` 分支开发完成后合并。
